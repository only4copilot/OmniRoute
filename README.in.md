## 🤔ओम्नीरूट क्यों?

**पैसा बर्बाद करना और सीमा पार करना बंद करें:**

- <img src="https://img.shields.io/badge/✗-e74c3c?style=flat-square" height="16"/> सदस्यता कोटा हर महीने अप्रयुक्त रूप से समाप्त हो जाता है
- <img src="https://img.shields.io/badge/✗-e74c3c?style=flat-square" height="16"/> दर सीमा आपको कोडिंग के बीच में रोक देती है
- <img src="https://img.shields.io/badge/✗-e74c3c?style=flat-square" height="16"/> महँगे API ($20-50/माह प्रति प्रदाता)
- <img src="https://img.shields.io/badge/✗-e74c3c?style=flat-square" height="16"/> प्रदाताओं के बीच मैन्युअल स्विचिंग

**OmniRoute इसका समाधान करता है:**

- ✅ **सब्सक्रिप्शन अधिकतम करें** - कोटा ट्रैक करें, रीसेट से पहले हर बिट का उपयोग करें
- ✅ **ऑटो फ़ॉलबैक** - सदस्यता → एपीआई कुंजी → सस्ता → निःशुल्क, शून्य डाउनटाइम
- ✅ **मल्टी-अकाउंट** - प्रति प्रदाता खातों के बीच राउंड-रॉबिन
- ✅ **यूनिवर्सल** - क्लाउड कोड, कोडेक्स, जेमिनी सीएलआई, कर्सर, क्लाइन, ओपनक्लॉ, किसी भी सीएलआई टूल के साथ काम करता है

**OMNI_टोकन_468**

<summary><b>💸 1. "मैं एक महंगी सदस्यता के लिए भुगतान करता हूं लेकिन फिर भी सीमा से बाधित होता हूं"</b></summary>

डेवलपर्स क्लाउड प्रो, कोडेक्स प्रो, या गिटहब कोपायलट के लिए $20-200/माह का भुगतान करते हैं। यहां तक ​​कि भुगतान करने पर भी, कोटा की एक सीमा होती है - 5 घंटे का उपयोग, साप्ताहिक सीमा, या प्रति मिनट की दर सीमा। मध्य-कोडिंग सत्र में, प्रदाता प्रत्युत्तर देना बंद कर देता है और डेवलपर प्रवाह और उत्पादकता खो देता है।

**ओम्नीरूट इसे कैसे हल करता है:**

## 🧪 मूल्यांकन (Evaluations)

ओमनीरूट में गोल्डन सेट के मुकाबले एलएलएम प्रतिक्रिया गुणवत्ता का परीक्षण करने के लिए एक अंतर्निहित मूल्यांकन ढांचा शामिल है। डैशबोर्ड में **एनालिटिक्स → इवेल्स** के माध्यम से इसे एक्सेस करें।

### बिल्ट-इन गोल्डन सेट

प्री-लोडेड "ओम्नीरूट गोल्डन सेट" में इसके लिए परीक्षण मामले शामिल हैं:

- नमस्ते, गणित, भूगोल, कोड जनरेशन
- JSON प्रारूप अनुपालन, अनुवाद, मार्कडाउन पीढ़ी
- सुरक्षा इनकार (हानिकारक सामग्री), गिनती, बूलियन तर्क

### मूल्यांकन रणनीतियाँ

| रणनीति     | विवरण                                             | उदाहरण                           |
| ---------- | ------------------------------------------------- | -------------------------------- |
| `exact`    | आउटपुट बिल्कुल मेल खाना चाहिए                     | `"4"`                            |
| `contains` | आउटपुट में सबस्ट्रिंग (केस-असंवेदनशील) होना चाहिए | `"Paris"`                        |
| `regex`    | आउटपुट रेगेक्स पैटर्न से मेल खाना चाहिए           | `"1.*2.*3"`                      |
| `custom`   | कस्टम जेएस फ़ंक्शन सही/गलत लौटाता है              | `(output) => output.length > 10` |

- `5h ON` + `Weekly OFF`: केवल 5 घंटे का उपयोग ही खाते को ब्लॉक कर सकता है।

### 🔐 रिमोट सर्वर पर OAuth

<a name="oauth-on-a-remote-server"></a>
<a name="oauth-em-servidor-remoto"></a>

> **⚠️ वीपीएस, डॉकर या किसी रिमोट सर्वर पर ओमनीरूट चलाने वाले उपयोगकर्ताओं के लिए महत्वपूर्ण**

#### एंटीग्रेविटी/जेमिनी सीएलआई ओएथ दूरस्थ सर्वर पर विफल क्यों होता है?

**एंटीग्रेविटी** और **जेमिनी सीएलआई** प्रदाता **Google OAuth 2.0** का उपयोग करते हैं। ऐप के Google क्लाउड कंसोल में पूर्व-पंजीकृत यूआरआई में से एक से सटीक मिलान करने के लिए Google को OAuth प्रवाह में `redirect_uri` की आवश्यकता होती है।

ओमनीरूट में बंडल किए गए OAuth क्रेडेंशियल **केवल `localhost` के लिए** पंजीकृत हैं। जब आप किसी दूरस्थ सर्वर (जैसे `https://omniroute.myserver.com`) पर ओमनीरूट तक पहुंचते हैं, तो Google प्रमाणीकरण को अस्वीकार कर देता है:

```
Error 400: redirect_uri_mismatch
```

#### समाधान: अपने स्वयं के OAuth क्रेडेंशियल कॉन्फ़िगर करें

आपको अपने सर्वर के यूआरआई के साथ Google क्लाउड कंसोल में एक **OAuth 2.0 क्लाइंट आईडी** बनाना होगा।

#### कदम दर कदम

**1. Google क्लाउड कंसोल खोलें**

यहां जाएं: [https://console.cloud.google.com/apis/credentials](https://console.cloud.google.com/apis/credentials)

**2. एक नया OAuth 2.0 क्लाइंट आईडी बनाएं**

- **"+ क्रेडेंशियल बनाएं"** → **"OAuth क्लाइंट आईडी"** पर क्लिक करें
- एप्लिकेशन प्रकार: **"वेब एप्लिकेशन"**
- नाम: कुछ भी जो आपको पसंद हो (जैसे `OmniRoute Remote`)

**3. अधिकृत रीडायरेक्ट यूआरआई जोड़ें**

**"अधिकृत रीडायरेक्ट यूआरआई"** फ़ील्ड में, जोड़ें:

```
https://your-server.com/callback
```

> `your-server.com` को अपने सर्वर के डोमेन या IP से बदलें (यदि आवश्यक हो तो पोर्ट शामिल करें, उदाहरण के लिए `http://45.33.32.156:20128/callback`)।

**4. क्रेडेंशियल सहेजें और कॉपी करें**

बनाने के बाद, Google **क्लाइंट आईडी** और **क्लाइंट सीक्रेट** दिखाएगा।

**5. पर्यावरण चर सेट करें**

आपके `.env` (या डॉकर पर्यावरण चर) में:

```bash
# For Antigravity:
ANTIGRAVITY_OAUTH_CLIENT_ID=your-client-id.apps.googleusercontent.com
ANTIGRAVITY_OAUTH_CLIENT_SECRET=GOCSPX-your-secret

# For Gemini CLI:
GEMINI_OAUTH_CLIENT_ID=your-client-id.apps.googleusercontent.com
GEMINI_OAUTH_CLIENT_SECRET=GOCSPX-your-secret
GEMINI_CLI_OAUTH_CLIENT_SECRET=GOCSPX-your-secret
```

**6. ओम्निरूट को पुनरारंभ करें**

```bash
# npm:
npm run dev

# Docker:
docker restart omniroute
```

**7. पुनः कनेक्ट करने का प्रयास करें**

डैशबोर्ड → प्रदाता → एंटीग्रेविटी (या जेमिनी सीएलआई) → OAuth

Google अब सही ढंग से `https://your-server.com/callback` पर रीडायरेक्ट करेगा।

यदि आप अभी अपना स्वयं का क्रेडेंशियल सेट नहीं करना चाहते हैं, तो आप अभी भी **मैन्युअल यूआरएल प्रवाह** का उपयोग कर सकते हैं:

1. ओमनीरूट Google प्राधिकरण URL खोलता है
2. अधिकृत करने के बाद, Google `localhost` पर रीडायरेक्ट करने का प्रयास करता है (जो रिमोट सर्वर पर विफल हो जाता है)

4. उस यूआरएल को ओमनीरूट कनेक्शन मोडल में दिखाए गए फ़ील्ड में पेस्ट करें
5. **"कनेक्ट"** पर क्लिक करें

> यह काम करता है क्योंकि यूआरएल में प्राधिकरण कोड इस बात पर ध्यान दिए बिना मान्य है कि रीडायरेक्ट पेज लोड किया गया है या नहीं।

<summary><b>🇧🇷 पुर्तगाली भाषा में वर्सो</b></summary>

#### क्या आप एंटीग्रेविटी/जेमिनी सीएलआई के लिए जिम्मेदार हैं?

प्रमाणित करने के लिए **एंटीग्रेविटी** और **मिथुन सीएलआई** का उपयोग करें **Google OAuth 2.0** का उपयोग करें। Google को यह देखना है कि `redirect_uri` का उपयोग बिना किसी प्रवाह के OAuth सेजा **exatamente** के माध्यम से किया गया है और आपको Google क्लाउड कंसोल के लिए आवेदन करने से पहले URIs की आवश्यकता है।

जैसा कि OAuth का प्रमाण है, कोई ओम्निरूट कैडस्ट्राड नहीं है ** `localhost`** के लिए एपेनास। एक सर्विडोर रिमोट (उदा: `https://omniroute.meuservidor.com`) पर ओम्नीरूट का उपयोग कैसे करें, या Google एक प्रमाणीकरण वापस लेता है:

```
Error 400: redirect_uri_mismatch
```

#### समाधान: OAuth को कॉन्फ़िगर करें

आपका सटीक विवरण **OAuth 2.0 क्लाइंट आईडी** आपके सर्वर पर यूआरआई के साथ Google क्लाउड कंसोल नहीं है।

#### पासो ए पासो

**1. Google क्लाउड कंसोल तक पहुंच**

अब्राहम: [https://console.cloud.google.com/apis/credentials](https://console.cloud.google.com/apis/credentials)

**2. नया OAuth 2.0 क्लाइंट आईडी देखें**

- उन्हें क्लिक करें **"+ क्रेडेंशियल बनाएं"** → **"OAuth क्लाइंट आईडी"**
- आवेदन टिप: **"वेब एप्लिकेशन"**
- नोम: एस्कोल्हा क्वेल्कर नोम (उदा: `OmniRoute Remote`)

**3. अधिकृत रीडायरेक्ट यूआरआई के रूप में एडिकियोन**

कोई शिकायत नहीं **"अधिकृत रीडायरेक्ट यूआरआई"**, आदि:

```
https://seu-servidor.com/callback
```

> स्थानापन्न `seu-servidor.com` अपने आईपी को अपने सर्वर पर रखें (इसमें एक आवश्यक पोर्ट भी शामिल है, उदाहरण के लिए: `http://45.33.32.156:20128/callback`)।

**4. साख के रूप में सहेजें और कॉपी करें**

एपोस क्रियर, Google द्वारा **क्लाइंट आईडी** और **क्लाइंट सीक्रेट**।

**5. परिवेश परिवर्तन** के रूप में कॉन्फ़िगर करें

कोई `.env` नहीं (आप डॉकर के परिवेश को कैसे बदल सकते हैं):

**OMNI_टोकन_50**

**6. ओम्निरूट का नवीनीकरण**

```bash
# Se usando npm:
npm run dev

# Se usando Docker:
docker restart omniroute
```

**7. नए सिरे से संपर्क करें**

डैशबोर्ड → प्रदाता → एंटीग्रेविटी (या जेमिनी सीएलआई) → OAuth

`https://seu-servidor.com/callback` के लिए Google पुनः निर्देशित और एक प्रमाणीकरण कार्य।

#### वर्कअराउंड टेम्पोरैरियो (सेम कॉन्फिगरेशन क्रेडेंशियल प्रोप्रियास)

यदि आप पहले से ही उचित क्रेडेंशियल प्राप्त नहीं करना चाहते हैं, तो आपके लिए फ्लक्सो का उपयोग करना संभव है **यूआरएल का मैनुअल**:

1. Google पर स्वचालित URL का उपयोग करके ओम्निरूट का उपयोग करें
2. ऑटो रिज़र्व के लिए एपोस, या `localhost` के लिए Google रीडायरेक्ट (यदि कोई सर्वर रिमोट नहीं है)
3. **एक यूआरएल को पूरा कॉपी करें** अपने ब्राउजर से दोबारा डाउनलोड करें (मुझे लगता है कि एक पेज अभी भी उपलब्ध है)
4. ओम्निरूट से जुड़ने के लिए कोई भी यूआरएल नहीं है
5. उन्हें क्लिक करें **"कनेक्ट"**

> यह समाधान यूआरएल को स्वचालित रूप से डाउनलोड करने के लिए काम कर रहा है और आपके द्वारा किए गए रीडायरेक्ट को स्वतंत्र रूप से वैध बनाता है।
